## Decisions Made

- **Parity-gate spot-check: Option B (trust stdlib golden-file pattern).** Task 2 offered the user Option A (temporarily edit `go/pkg/types/classes.go` THAC0, confirm TestGolden fails with a diff, revert) vs Option B (skip the spot-check and rely on the well-established stdlib golden-file design). User approved without requesting Option A. Rationale: the pattern is battle-tested; the design-level argument that any byte difference in the rendered snapshot produces a `t.Errorf` diff is self-evident from `golden_test.go`. Cost of Option A (edit/revert round-trip, risk of accidentally committing the mutation) exceeded its evidentiary value.
- **No pre-commit hook for `-update`.** Per RESEARCH Pitfall #6 and threat T-01-03-01, silent regeneration is mitigated via commit-message visibility (`-update` always produces a diff on `entities.golden` visible in any PR) rather than tooling. Hook deferred to a later phase if the team chooses.
- **Dedicated commit for the snapshot.** Plan 03 committed only `entities.golden` — no other file additions, no mixed concerns — so the baseline has a single point in git history that future phases can reference by SHA (`a7f4089`).

Plan 02 made these adjustments to the fixture runner during the skeleton build, and they shaped the snapshot output this plan committed:

- All three adjustments landed in `go/pkg/golden/fixture.go` in Plan 02 commits (`7140dd6`, `4095712`) — no re-edit was needed in Plan 03

## Important Reviewer Observations (surface to subsequent phases)

These are structural observations from the committed snapshot that downstream plans should know about:

- **Race struct has no Imm/Res/Vuln fields.** Every race entry shows `Imm=- Res=- Vuln=-` because the current `types.Race` struct does not carry these traits (they live on affects/classes instead). When Phase 7/8 migrate vampire innate fire/silver vulnerability, these columns will populate, and the diff will be visible here. This is the expected baseline; do NOT "fix" the dashes.
- **Vampire is a class, not a race.** The RESEARCH Pitfall #7 warning about "Vampire race must surface fire/silver Vuln" was written against an older assumption. The current code models vampire as a class (Class=vampire in the class section); the race list contains no vampire entry. Migration planning should treat vampire vulnerability as a class-level trait.
- **`dMana=0` on every spell line.** All 7 spell fixtures show `dMana=0`, meaning the fixture runner bypasses mana cost deduction (likely casts via `Spell.Func` directly rather than through the full `DoCast` pipeline). This is a coverage gap to widen in a later phase if mana-cost drift becomes a concern, but it is the intended Phase 1 scope.
- **Backstab and kick miss on seed-42 first invocation.** The skill section shows `damage=0` for both Backstab and Kick. This is not a bug — it is the deterministic outcome of seed=42 on the first RNG draw. The value of the snapshot is that these numbers are FROZEN; if a future refactor changes `combat.Dice()` or the skill success formula, the damage figure will shift, the test will fail, and the migration author will have to justify the change.
